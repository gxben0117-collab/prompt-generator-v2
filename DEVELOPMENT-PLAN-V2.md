# 出圖自組咒語生產器 v2.0 開發計畫書

**撰寫日期**: 2026-06-13  
**目標**: 模組化重構，保留所有核心資產，提升可維護性與擴展性

## 一、專案現況分析

### 1.1 核心優勢資產（必須 100% 保留）

#### 🎯 咒語鎖臉系統（核心競爭力）
- **功能**: 保留上傳真人原始五官特徵
- **實現**: `makeup` 欄位的精準描述語言
- **價值**: 市場獨有，用戶最重視的功能
- **狀態**: 已在 2,175 張角色卡中驗證有效

#### 👗 10 層服裝系統
- **結構**: `costumeLayer1` ~ `costumeLayer10`
- **邏輯**: 從內襯 → 胸腰 → 主體 → 外層 → 配件 → 頭飾 → 最終輪廓
- **價值**: 建立真實可穿戴的服裝物理邏輯
- **狀態**: 已完整定義並在所有角色卡中使用

#### 🎨 36 個父分類的獨特特色知識
每個分類都有：
- 視覺風格定義（色彩、材質、光影）
- 服裝層次偏好（哪些 layer 優先）
- 動作姿態規則
- 場景道具邏輯

**範例**：
- **仙俠神話**: 粉金色、披帛、踏蓮動作、宮燈光影
- **暗黑王族**: 暗紫色、戰袍、王座邊緣、月光透窗
- **巴黎咖啡館**: 奶油白、風衣、端咖啡、窗光

#### 📦 2,175 張角色卡資料
- 當前保留: 2,175 張（已刪除 27 張）
- 格式: 完整定義物件
- 價值: 內容資產，直接變現

### 1.2 當前架構問題

#### 問題 1: 分類特色知識散落各處
```javascript
// 目前狀態：分類知識分散在
// 1. categoryClassifier.js 的 keywords
// 2. promptEngine.js 的 inferCategoryVisualModeText
// 3. profileFactory.js 的硬編碼邏輯
// 4. 每張角色卡的 sceneAction 裡

// 結果：很難一眼看出「仙俠神話」的完整特色是什麼
```

#### 問題 2: 角色卡資料與邏輯耦合
- 2,175 張角色卡直接寫在 `data.js` 中（7,000+ 行）
- 修改一張卡需要重新編譯整個專案
- 難以批次更新或外部管理

#### 問題 3: 服裝層次選擇邏輯不清晰
```javascript
// 目前：hardcoded 在 profileFactory.js
const highlightLayers = [2, 4, 7, 9]; // 為什麼是這些？

// 應該：每個分類明確定義
仙俠神話.highlightLayers = [2, 4, 7, 9]; // 飄逸、神性層
暗黑王族.highlightLayers = [3, 6, 8];    // 戰袍、披風、珠寶
```

#### 問題 4: 題材化邏輯重複
- `布料動態` 邏輯散落在多處
- `highlightLayers` 每次重新計算
- 沒有統一的「分類特色模組」

## 二、v2.0 架構設計

### 2.1 核心原則

✅ **保留精華**: 咒語鎖臉、10 層服裝、36 個分類特色、2,175 張角色卡  
✅ **模組化**: 分類特色集中管理  
✅ **外部化**: 角色卡資料可獨立更新  
✅ **可擴展**: 新增分類或角色卡不需改核心邏輯  

### 2.2 新架構模組圖

```
src/
├── core/
│   ├── faceLockEngine.js       # 咒語鎖臉核心（提取自 makeup 邏輯）
│   ├── costumeLayerEngine.js   # 10 層服裝系統（保留現有邏輯）
│   └── promptBuilder.js        # 主 prompt / 增強 / 純咒語 三段式輸出
│
├── categories/
│   ├── index.js                # 分類註冊表
│   ├── 仙俠神話.js             # 每個分類一個模組
│   ├── 暗黑王族.js
│   ├── 巴黎咖啡館.js
│   └── ...（36 個分類）
│
├── profiles/
│   ├── loader.js               # 角色卡載入器
│   ├── validator.js            # 角色卡驗證
│   └── data/                   # 外部化角色卡資料（JSON）
│       ├── 仙俠神話.json       # 126 張
│       ├── 暗黑王族.json       # 61 張
│       └── ...
│
├── engines/
│   ├── categoryInference.js   # 分類推斷（保留現有邏輯）
│   ├── visualModeEngine.js    # 視覺模式推斷
│   └── fabricMotionEngine.js  # 布料動態題材化
│
└── ui/
    └── （保持不變）
```

### 2.3 分類模組標準格式

每個分類都是一個獨立模組，完整定義其特色：

```javascript
// src/categories/仙俠神話.js
export const 仙俠神話分類 = {
  id: "仙俠神話",
  displayName: "仙俠神話 / 古裝陸劇",
  
  // 視覺風格
  visualStyle: {
    defaultColors: ["粉金", "寶石藍", "仙氣白"],
    defaultVisualMode: "Netflix 東方奇幻",
    lightingKeywords: ["宮燈", "燭火", "金箔反射"],
  },
  
  // 服裝層次偏好
  costumePreferences: {
    highlightLayers: [2, 4, 7, 9],  // 飄逸、神性層
    defaultFabricMotion: "大動態飄紗",
    keyElements: ["披帛", "外紗", "流雲斗篷", "extra-long flowing silk drapery"],
  },
  
  // 動作姿態
  actionPatterns: {
    preferred: ["踏蓮", "雲階", "持法器", "扶石柱回身"],
    avoid: ["端咖啡", "翻書", "手持手機"],  // 現代元素
  },
  
  // 道具規則
  propRules: {
    allowed: ["法器", "權杖", "聖杯", "仙氣披帛"],
    context: "儀式場景、神殿、雲階",
  },
  
  // 分類推斷規則（保留現有 keywords）
  inference: {
    keywords: ["仙俠", "神話", "修真", "宮廷", "古裝"],
    exclude: ["現代", "都市", "賽博"],
  },
  
  // 咒語鎖臉規則（每個分類可微調）
  faceLockRules: {
    baseMakeup: "東方電影妝感",
    preserveFeatures: ["臉型", "眼型", "鼻型", "下顎線"],
    avoidTemplates: ["AI 仙女模板", "AI 美女模板"],
  },
  
  // 現有角色卡數量
  profileCount: 126,
};
```

## 三、實施計畫

### 3.1 階段劃分（總計 5-7 天）

#### 階段 1: 資產盤點與提取（1 天）
**目標**: 完整提取現有精華邏輯

**任務**:
1. 提取咒語鎖臉規則 → `faceLockEngine.js`
2. 提取 10 層服裝邏輯 → `costumeLayerEngine.js`
3. 提取 36 個分類的完整特色知識
4. 自動掃描 2,175 張角色卡，提取每個分類的：
   - 最常用色彩組合（top 10）
   - 最常用服裝關鍵字（詞頻統計）
   - 最常用動作模式（pattern 分析）
   - 最常用光影描述（cluster 分析）

**產出**:
- `ASSET-INVENTORY.md`（資產清單報告）
- `CATEGORY-ANALYSIS.json`（36 個分類的統計分析）

#### 階段 2: 分類模組建立（2 天）
**目標**: 為 36 個分類建立標準模組

**任務**:
1. 建立分類模組模板
2. 為每個分類填寫完整定義：
   - 視覺風格（來自統計分析）
   - 服裝偏好（來自現有 profileFactory 邏輯）
   - 動作規則（來自 sceneAction 分析）
   - 道具邏輯（來自 prop 統計）
   - 推斷規則（保留現有 categoryClassifier 邏輯）
3. 建立分類註冊系統

**產出**:
- `src/categories/` 目錄下 36 個分類模組
- `src/categories/index.js` 註冊表

#### 階段 3: 核心引擎重寫（2 天）
**目標**: 重寫核心邏輯，使用新的分類模組

**任務**:
1. 重寫 `promptBuilder.js`：
   - 使用分類模組的 `visualStyle`
   - 使用分類模組的 `costumePreferences`
   - 保留三段式輸出邏輯
2. 重寫 `categoryInference.js`：
   - 使用分類模組的 `inference` 規則
   - 保留現有推斷算法
3. 重寫 `visualModeEngine.js`：
   - 基於分類模組的 `visualStyle`
   - 保留現有 visualMode 邏輯
4. 重寫 `fabricMotionEngine.js`：
   - 基於分類模組的 `costumePreferences.defaultFabricMotion`
   - 保留題材化邏輯

**產出**:
- 新的 `src/engines/` 目錄
- 單元測試通過（移植現有測試）

#### 階段 4: 角色卡遷移與驗證（1 天）
**目標**: 將 2,175 張角色卡遷移到新格式

**任務**:
1. 編寫遷移腳本：
   - 讀取現有 data.js
   - 按分類分組
   - 輸出為 JSON 格式
2. 執行遷移並驗證：
   - 確認所有角色卡都被遷移
   - 確認資料完整性
   - 建立 loader 和 validator
3. 整合測試：
   - 抽樣 100 張角色卡測試輸出
   - 比對新舊系統輸出一致性

**產出**:
- `src/profiles/data/` 目錄下 36 個 JSON 檔案
- 遷移驗證報告
- 100 張角色卡測試報告

#### 階段 5: UI 整合與測試（1 天）
**目標**: 整合新架構到現有 UI

**任務**:
1. 更新 UI 呼叫邏輯
2. 全面測試：
   - 測試所有 36 個分類
   - 測試隨機 200 張角色卡
   - 測試三段式輸出
   - 測試咒語鎖臉功能
3. 效能測試：
   - 確認載入時間 < 2 秒
   - 確認生成速度未降低

**產出**:
- 完整的整合測試報告
- 效能基準測試報告
- Bug 修復清單

## 四、風險評估與應對

### 4.1 風險矩陣

| 風險 | 機率 | 影響 | 應對策略 |
|------|------|------|----------|
| 咒語鎖臉邏輯遺失 | 低 | 極高 | 階段 1 完整提取並驗證 |
| 分類特色知識不完整 | 中 | 高 | 使用自動統計 + 人工審核 |
| 角色卡遷移錯誤 | 中 | 高 | 編寫驗證腳本，100% 校驗 |
| 效能下降 | 低 | 中 | 效能測試，優化載入邏輯 |
| 現有測試無法通過 | 中 | 中 | 移植並調整測試案例 |

### 4.2 回退計畫

- **每個階段完成後**: Git commit + tag
- **保留現有系統**: v1.0 分支不刪除
- **並行部署**: v2.0 測試期間 v1.0 仍可使用

## 五、預期成果

### 5.1 技術成果

✅ **更清晰的架構**

- 10 層服裝：100% 保留
